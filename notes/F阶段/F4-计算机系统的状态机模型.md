## 二、处理器执行程序的基本过程

计算机执行程序时，并不是直接理解 C 语言，而是执行一条条机器指令。

处理器执行程序的核心过程可以概括为：

```text
取指令
执行指令
更新 PC
```

其中：

- `PC`：程序计数器，表示当前或下一条要执行的指令位置
- `GPR`：通用寄存器，用来保存临时数据
- `指令`：告诉处理器要执行什么操作
- `ISA`：指令集架构，规定每条指令的含义和执行效果

在 F4 文档中的简单 ISA 模型里，有 4 个通用寄存器：

```text
r0, r1, r2, r3
```

处理器状态可以表示为：

```text
(PC, r0, r1, r2, r3)
```

每执行一条指令，这个状态就会按照指令语义发生一次转移。

## 三、官方数列求和程序

官方给出的程序用于计算：

```text
1 + 2 + 3 + ... + 10
```

指令序列如下：

```text
0: li r0, 10
1: li r1, 0
2: li r2, 0
3: li r3, 1
4: add r1, r1, r3
5: add r2, r2, r1
6: bner0 r1, 4
7: bner0 r3, 7
```

各寄存器含义：

| 寄存器 | 作用         |
| ------ | ------------ |
| r0     | 保存上限 10  |
| r1     | 当前加到的数 |
| r2     | 累加和 sum   |
| r3     | 常数 1       |

循环部分是：

```text
4: add r1, r1, r3
5: add r2, r2, r1
6: bner0 r1, 4
```

含义是：

```text
r1 = r1 + 1
r2 = r2 + r1
如果 r1 != r0，则跳回 PC=4 继续循环
```

## 五、10 以内奇数求和程序

接着，我尝试使用相同的 sISA 指令计算：

```text
1 + 3 + 5 + 7 + 9
```

目标结果是：

```text
25
```

设计思路：

| 寄存器 | 作用       |
| ------ | ---------- |
| r0     | 终止值 11  |
| r1     | 当前奇数   |
| r2     | 累加和 sum |
| r3     | 常数 2     |

这里 `r0` 设置为 11，而不是 9。

原因是程序每轮先加当前奇数，再让当前奇数加 2。加完 9 后，`r1` 会变成 11，此时停止循环。

程序如下：

```text
0: li r0, 11
1: li r1, 1
2: li r2, 0
3: li r3, 2
4: add r2, r2, r1
5: add r1, r1, r3
6: bner0 r1, 4
7: bner0 r3, 7
```

循环部分：

```text
4: add r2, r2, r1
5: add r1, r1, r3
6: bner0 r1, 4
```

含义是：

```text
sum = sum + 当前奇数
当前奇数 = 当前奇数 + 2
如果当前奇数 != 11，则继续循环
```

状态变化如下：

| PC   | r0   | r1   | r2   | r3   | 说明       |
| ---- | ---- | ---- | ---- | ---- | ---------- |
| 0    | 0    | 0    | 0    | 0    | 初始       |
| 1    | 11   | 0    | 0    | 0    | r0 = 11    |
| 2    | 11   | 1    | 0    | 0    | r1 = 1     |
| 3    | 11   | 1    | 0    | 0    | r2 = 0     |
| 4    | 11   | 1    | 0    | 2    | r3 = 2     |
| 5    | 11   | 1    | 1    | 2    | 加 1       |
| 6    | 11   | 3    | 1    | 2    | r1 变成 3  |
| 4    | 11   | 3    | 1    | 2    | 跳回       |
| 5    | 11   | 3    | 4    | 2    | 加 3       |
| 6    | 11   | 5    | 4    | 2    | r1 变成 5  |
| 4    | 11   | 5    | 4    | 2    | 跳回       |
| 5    | 11   | 5    | 9    | 2    | 加 5       |
| 6    | 11   | 7    | 9    | 2    | r1 变成 7  |
| 4    | 11   | 7    | 9    | 2    | 跳回       |
| 5    | 11   | 7    | 16   | 2    | 加 7       |
| 6    | 11   | 9    | 16   | 2    | r1 变成 9  |
| 4    | 11   | 9    | 16   | 2    | 跳回       |
| 5    | 11   | 9    | 25   | 2    | 加 9       |
| 6    | 11   | 11   | 25   | 2    | r1 变成 11 |
| 7    | 11   | 11   | 25   | 2    | 停止       |

最终：

```text
r2 = 25
```

也就是：

```text
1 + 3 + 5 + 7 + 9 = 25
```

## 六、状态机的基本概念

状态机一般包含：

```text
状态集合
激励事件集合
状态转移规则
初始状态
```

可以简单理解为：

```text
当前状态在某个事件触发下，按照规则转移到下一状态。
```

在 F4 中，可以把计算机系统理解为状态机。

对于 ISA 状态机：

| 状态机要素   | ISA 中的对应内容                  |
| ------------ | --------------------------------- |
| 状态         | PC、GPR、内存                     |
| 激励事件     | 执行一条指令                      |
| 状态转移规则 | 指令语义                          |
| 初始状态     | 程序开始前的 PC、寄存器、内存状态 |

也就是说，处理器执行程序，本质上就是：

```text
按照指令语义，不断更新 PC、寄存器和内存状态
```

每条指令规定了一次状态转移。

## 七、C 程序状态机

C 程序也可以看作状态机。

例如：

```c
int x = 1;
int y = 2;
int z = x + y;
printf("z = %d\n", z);
```

在 C 程序中，状态可以理解为：

```text
变量的值 + 当前执行位置
```

如果写成状态形式，可以表示为：

```text
(PC, x, y, z)
```

这里的 PC 不是 CPU 里的真实 PC，而是为了理解程序执行过程抽象出来的“当前执行到哪一条语句”。

C 程序状态机中：

| 状态机要素   | C 程序中的对应内容       |
| ------------ | ------------------------ |
| 状态         | 变量的值 + 当前执行位置  |
| 激励事件     | 执行一条语句             |
| 状态转移规则 | C 语言语句的语义         |
| 初始状态     | 程序开始时变量和执行位置 |

所以，C 程序执行过程本质上也是状态不断变化的过程。

## 八、数字电路状态机

数字电路也可以看作状态机。

在数字电路中：

| 状态机要素   | 数字电路中的对应内容               |
| ------------ | ---------------------------------- |
| 状态         | 寄存器、触发器等时序逻辑保存的信息 |
| 激励事件     | 时钟上升沿                         |
| 状态转移规则 | 组合逻辑计算出的下一状态           |
| 初始状态     | 复位后的状态                       |

也就是说：

```text
时序逻辑保存当前状态
组合逻辑计算下一状态
时钟让状态更新
```

例如 F3 中做过的 4 位计数器：

```text
Q_next = Q + 1
```

当前状态是：

```text
Q = 0011
```

组合逻辑计算下一状态：

```text
Q_next = 0100
```

时钟上升沿到来后：

```text
Q = 0100
```

所以计数器的状态转移为：

```text
0000 → 0001 → 0010 → 0011 → ... → 1111 → 0000
```

## 九、用状态机理解数列求和电路

F3 中实现过数列求和电路：

```text
1 + 2 + ... + 10
```

电路中有两个核心寄存器：

```text
cnt：当前加到几
sum：当前累加和
```

所以它的状态可以写成：

```text
(cnt, sum)
```

初始状态：

```text
(0, 0)
```

组合逻辑计算：

```text
cnt_next = cnt + 1
sum_next = sum + cnt_next
```

每来一个时钟上升沿，状态更新一次：

| 当前状态 `(cnt, sum)` | 下一状态   |
| --------------------- | ---------- |
| `(0, 0)`              | `(1, 1)`   |
| `(1, 1)`              | `(2, 3)`   |
| `(2, 3)`              | `(3, 6)`   |
| `(3, 6)`              | `(4, 10)`  |
| `...`                 | `...`      |
| `(9, 45)`             | `(10, 55)` |

这说明，数字电路和 ISA 程序一样，也是在做状态转移。

只是 ISA 状态通常写成：

```text
(PC, r0, r1, r2, r3)
```

而数字电路状态可以写成：

```text
(cnt, sum)
```

但它们的核心思想相同：

```text
当前状态 → 根据规则计算下一状态 → 更新状态
```

## 十、用状态机理解电子时钟

F3 中还实现了电子时钟，显示格式为：

```text
MM:SS
```

也就是：

```text
分钟十位 分钟个位 : 秒钟十位 秒钟个位
```

所以电子时钟的状态可以写成：

```text
(min_tens, min_ones, sec_tens, sec_ones)
```

例如：

```text
00:00 → (0, 0, 0, 0)
00:09 → (0, 0, 0, 9)
00:10 → (0, 0, 1, 0)
01:00 → (0, 1, 0, 0)
59:59 → (5, 9, 5, 9)
```

初始状态：

```text
(0, 0, 0, 0)
```

激励事件：

```text
Clock 上升沿
```

状态转移规则：

```text
sec_ones 每秒加 1；
当 sec_ones 等于 9 时，下一状态中 sec_ones 回到 0，同时 sec_tens 加 1；
当 sec_tens 等于 5 且 sec_ones 等于 9 时，下一状态中秒变为 00，同时 min_ones 加 1；
当 min_ones 等于 9 且秒为 59 时，下一状态中 min_ones 回到 0，同时 min_tens 加 1；
当当前状态为 59:59 时，下一状态回到 00:00；
如果 CLR=1，则下一次时钟后状态直接变为 00:00。
```

所以电子时钟本质上也是一个数字电路状态机：

```text
寄存器保存当前时间
比较器判断是否到达 9、5、59:59
进位逻辑决定下一状态
Clock 上升沿更新状态
```

## 十一、编译器的作用

计算机不能直接执行 C 代码。

例如：

```c
sum = sum + i;
```

CPU 并不能直接理解这条 C 语句。

CPU 能执行的是 ISA 指令，例如：

```text
add r2, r2, r1
```

所以，想让计算机执行 C 程序，就必须先把 C 程序翻译成 CPU 能执行的指令序列。

这个过程叫：

```text
编译
```

完成这个工作的程序叫：

```text
编译器
```

从状态机角度看：

```text
编译器 = 把 C 程序状态机翻译成 ISA 状态机
```

也就是说：

| C 程序             | ISA            |
| ------------------ | -------------- |
| 变量               | GPR 或内存     |
| C 程序中的执行位置 | ISA 中的 PC    |
| C 语句             | 一条或多条指令 |
| 语句语义           | 指令序列语义   |

例如：

```c
sum = sum + i;
```

如果约定：

```text
sum 放在 r2
i 放在 r1
```

那么可以翻译为：

```text
add r2, r2, r1
```

其效果为：

```text
r2 = r2 + r1
```

刚好对应 C 语句：

```text
sum = sum + i
```

## 十二、CPU 设计的作用

编译器解决的是：

```text
C 程序 → ISA 指令序列
```

CPU 设计解决的是：

```text
ISA 指令序列 → 数字电路执行
```

也就是说：

```text
CPU 设计 = 用数字电路状态机实现 ISA 状态机
```

例如指令：

```text
add r2, r2, r1
```

ISA 层面规定它的语义是：

```text
R[2] = R[2] + R[1]
PC = PC + 1
```

硬件实现时，就需要：

```text
从寄存器堆读出 r2
从寄存器堆读出 r1
送入 ALU 做加法
将结果写回 r2
PC 更新为下一条指令
```

所以 CPU 中需要：

- PC：保存当前指令地址
- 指令存储器：保存程序指令
- GPR：保存通用寄存器
- ALU：执行加法等运算
- MUX：选择数据来源
- 控制逻辑：根据指令决定电路如何工作
- Clock：推动状态更新

## 十三、C 程序、ISA、数字电路的统一关系

三层关系可以总结为：

```text
C 程序
↓ 编译器
ISA 指令序列
↓ CPU 硬件
数字电路状态机
```

对应关系：

| 层次     | 状态                    | 状态转移规则    | 实现者         |
| -------- | ----------------------- | --------------- | -------------- |
| C 程序   | 变量 + 当前执行位置     | C 语句语义      | 程序员写代码   |
| ISA      | PC + GPR + 内存         | 指令语义        | 编译器生成指令 |
| 数字电路 | 寄存器 / 触发器保存的值 | 组合逻辑 + 时钟 | CPU 硬件       |

可以这样理解：

```text
C 程序中的 sum、i
↓
ISA 中的 r2、r1、PC
↓
数字电路中的寄存器、ALU、MUX、控制信号、Clock
```

程序真正运行时，本质是：

```text
编译出的指令序列控制 CPU 电路进行一次又一次状态转移
```

## 十四、“机器永远是对的”的理解

F4 中很重要的一点是：

```text
机器永远是对的
```

更准确地说：

```text
计算机系统的行为是按照规则精确发生的。
如果现象和预期不同，通常是我们对状态、指令语义或转移规则理解不完整。
```

后面调试程序或处理器时，不应该只凭感觉判断对错，而应该问：

```text
当前状态是什么？
执行了什么指令？
这条指令的语义是什么？
下一状态应该是什么？
实际状态哪里不一样？
```

这个思想对后面学习 PA、NEMU 和 CPU 调试非常重要。

## 十五、F4 阶段总结

F4 阶段的核心是用状态机模型统一理解 C 程序、ISA 和数字电路。

C 程序可以看作变量和当前执行位置组成的状态机，执行每条语句都会让程序状态发生变化。

ISA 可以看作由 PC、GPR 和内存组成的状态机，每条指令的语义规定了状态如何变化。

数字电路也可以看作状态机，其中寄存器和触发器保存当前状态，组合逻辑计算下一状态，时钟触发状态更新。

编译器的作用是把 C 程序状态机翻译成 ISA 指令序列；CPU 设计的作用是用数字电路状态机实现 ISA 规定的状态转移。

因此，程序执行的本质就是：

```text
指令序列控制 CPU 不断进行状态转移
```
