# 问题记录：F5 sCPU 设计过程中的问题与解决

## 一、问题概况

在 F5 阶段设计支持数列求和的简单处理器 sCPU 时，主要遇到了以下几类问题：

- Logisim 多项目导入导致子电路依赖混乱
- `reg8` 无法正常写入
- `instr` 不知道如何观察
- `gpr4x8` 封装时接口设计不清楚
- 8 位输出误以为需要 8 个 Pin
- `bner0` 分支指令不跳转
- 比较器输出逻辑理解错误
- PC 为什么最后停在 7

这些问题都和处理器设计中的模块封装、数据通路、控制信号、状态更新有关。

------

## 二、问题一：多项目导入导致 reg8 无法正常使用

### 现象

在 sCPU 项目中直接导入之前 F3 数字时钟项目后，发现里面的 `reg8` 无法正常工作。

具体表现为：

```text
reg8 内部调用的 reg4 出现异常
寄存器输出一直是红色 E
无法正常写入数据
```

后来发现，数字时钟项目中的 `reg8` 依赖另一个 F3 项目里的 `reg4`。当前 sCPU 项目如果直接导入数字时钟项目，再单独导入包含 `reg4` 的项目，就容易出现库依赖混乱。

------

### 原因分析

原来的依赖关系比较复杂：

```text
sCPU 项目
├── digital_clock 项目
│   └── reg8
│       └── 依赖另一个项目里的 reg4
└── 另外导入的 reg4 项目
```

Logisim 在多个 `.circ` 项目互相导入时，如果存在同名子电路或嵌套依赖，容易出现：

```text
模块名字冲突
子电路引用路径变化
原来的 reg4 找不到
reg8 内部依赖丢失
```

因此，虽然表面上 `reg8` 还在，但它内部调用的 `reg4` 可能已经不是原来的那个，导致寄存器无法正常工作。

------

### 解决方法

不要在 F5 项目中直接导入整个 F3 数字时钟项目。

改为在当前 sCPU 项目中重新整理基础模块：

```text
reg4
reg8
adder4
adder8
gpr4x8
```

最终在当前 sCPU 项目中重新搭建 `reg8`，使它直接调用当前项目中的 `reg4`。

这样依赖关系变成：

```text
sCPU 当前项目
├── reg4
├── reg8
├── gpr4x8
└── sCPU_main
```

结构清楚后，`reg8` 可以正常写入。

------

### 经验总结

F5 阶段不要再直接导入 F3 的综合项目，例如：

```text
digital_clock
sequence_sum
counter_mod10
counter_mod6
```

这些项目内部依赖较多，容易污染当前项目。

F5 最好只保留最基础、最稳定的模块。

------

## 三、问题二：寄存器输出红色 E

### 现象

在测试 `reg8` 或 `gpr4x8` 时，寄存器输出端显示红色 `E`。

------

### 原因分析

红色 `E` 通常表示：

```text
寄存器当前状态未知
```

寄存器属于时序逻辑，如果没有经过一次有效的时钟上升沿写入确定值，它的输出可能就是未知状态。

------

### 解决方法

测试寄存器时，必须给出一次完整的时钟上升沿：

```text
CLK: 0 → 1
```

测试步骤：

```text
1. CLK = 0
2. EN = 1
3. 设置 D 输入为确定值
4. 将 CLK 从 0 切换到 1
5. 观察 Q 输出
```

例如：

```text
EN = 1
D = 00000111
CLK 0 → 1
```

预期：

```text
Q = 00000111
```

------

### 经验总结

寄存器不会因为输入变化就立刻更新，只有在有效时钟边沿到来时才写入数据。

组合逻辑是“输入变，输出马上变”。

时序逻辑是“时钟到，状态才更新”。

------

## 四、问题三：不知道如何观察 instr

### 现象

ROM 已经根据 PC 输出指令，但一开始不知道应该接什么东西观察 `instr[7:0]`。

------

### 解决方法

使用 `Probe` 观察 ROM 的 8 位输出。

连接方式：

```text
PC[3:0] → ROM Address
ROM Data[7:0] → Probe
```

在 Probe 属性中设置：

```text
Data Bits = 8
Radix = 16
```

这样就可以直接看到：

```text
8A
90
A0
B1
17
29
D1
DF
```

------

### 经验总结

观察总线时，不需要拆成一位一位看。

对于 8 位数据，推荐直接用 8 位 Probe，并设置显示进制为十六进制。

------

## 五、问题四：gpr4x8 封装时不知道接口怎么设计

### 现象

一开始只把 4 个 `reg8` 放在主电路里，没有明确寄存器堆应该有哪些输入和输出。

同时担心 `r0/r1/r2/r3` 都是 8 位输出，需要很多 Pin，电路会很乱。

------

### 解决方法

将 4 个 8 位寄存器封装为 `gpr4x8`。

最终接口设计为：

输入：

```text
CLK
wen
waddr[1:0]
wdata[7:0]
raddr1[1:0]
raddr2[1:0]
```

输出：

```text
rdata1[7:0]
rdata2[7:0]
r0[7:0]
r1[7:0]
r2[7:0]
r3[7:0]
```

其中：

```text
wen    表示是否写寄存器
waddr  表示写哪个寄存器
wdata  表示写入的数据
raddr1 表示读端口 1 选择哪个寄存器
raddr2 表示读端口 2 选择哪个寄存器
```

------

### 经验总结

Logisim 的 Pin 可以设置位宽。

一个 8 位输出不需要 8 个 Pin，只需要一个：

```text
Data Bits = 8
```

的 Output Pin。

因此：

```text
r0[7:0]
```

只需要一个 8 位 Output Pin。

------

## 六、问题五：gpr4x8 一开始缺少 wen

### 现象

最开始封装 `gpr4x8` 时，只用 `waddr` 选择写哪个寄存器，默认每条指令都会写寄存器。

这样在 `li` 阶段没有问题，但后面加入 `bner0` 后会出错。

------

### 原因分析

不是所有指令都需要写寄存器。

```text
li    需要写寄存器
add   需要写寄存器
bner0 不需要写寄存器
```

如果没有 `wen` 控制，那么执行 `bner0` 时也可能错误写入寄存器。

------

### 解决方法

在 `gpr4x8` 中加入写使能输入：

```text
wen
```

写入逻辑：

```text
r0.EN = wen & rd_is_r0
r1.EN = wen & rd_is_r1
r2.EN = wen & rd_is_r2
r3.EN = wen & rd_is_r3
```

如果 2-4 Decoder 自带 Enable，可以直接：

```text
wen → Decoder.Enable
```

然后 Decoder 输出分别接四个寄存器的 EN。

------

### 经验总结

寄存器堆不能只知道“写哪里”，还必须知道“这条指令到底写不写”。

这就是控制信号的作用。

------

## 七、问题六：add 指令需要双读端口

### 现象

实现 `add rd, rs1, rs2` 时，发现只靠 `r0/r1/r2/r3` 的调试输出还不够，需要根据指令字段选择两个源寄存器。

------

### 原因分析

`add` 指令格式为：

```text
00 rd rs1 rs2
```

它需要同时读出：

```text
R[rs1]
R[rs2]
```

然后相加：

```text
R[rd] = R[rs1] + R[rs2]
```

所以寄存器堆必须有两个读端口。

------

### 解决方法

在 `gpr4x8` 内部加入两个 4 选 1 的 8 位 MUX。

读端口 1：

```text
r0, r1, r2, r3 → 4选1 MUX → rdata1
raddr1 → MUX Select
```

读端口 2：

```text
r0, r1, r2, r3 → 4选1 MUX → rdata2
raddr2 → MUX Select
```

最终：

```text
rdata1 = R[raddr1]
rdata2 = R[raddr2]
```

------

### 经验总结

寄存器堆一般需要：

```text
一个写端口
两个读端口
```

这样才能支持类似：

```text
R[rd] = R[rs1] + R[rs2]
```

的指令。

------

## 八、问题七：bner0 不跳转，PC 继续加 1

### 现象

执行到：

```text
6: bner0 r1, 4
```

时，原本应该跳回：

```text
PC = 4
```

但实际 PC 继续变成：

```text
PC = 7
```

------

### 正确逻辑

`bner0` 指令格式：

```text
11 addr rs2
```

功能：

```text
if (R[0] != R[rs2])
    PC = addr;
else
    PC = PC + 1;
```

对于：

```text
D1 = 11010001
```

拆分为：

```text
opcode = 11
addr   = 0100
rs2    = 01
```

也就是：

```text
bner0 r1, 4
```

当时寄存器状态为：

```text
r0 = 10
r1 = 1
```

所以：

```text
r0 != r1
```

条件成立，PC 应该跳回 4。

------

### 原因分析

现场搭建的比较器输出逻辑是：

```text
相同 → 0
不同 → 1
```

这个输出本身已经表示：

```text
neq
```

但是在连接 `branch_taken` 时，又对比较器输出取了一次反。

导致：

```text
r0 != rdata2 时，compare_out = 1
取反后 neq = 0
```

于是：

```text
branch_taken = is_bne & neq
             = 1 & 0
             = 0
```

所以 PC 没有跳转，而是继续：

```text
PC = PC + 1
```

------

### 解决方法

去掉多余的 NOT。

正确连接：

```text
branch_taken = is_bne & compare_out
```

其中 `compare_out` 表示：

```text
r0 != rdata2
```

------

### 经验总结

比较器一定要先明确输出含义。

如果比较器定义为：

```text
相同 → 1
不同 → 0
```

那么它输出的是：

```text
equal
```

此时：

```text
neq = ~equal
```

如果比较器定义为：

```text
相同 → 0
不同 → 1
```

那么它输出的就是：

```text
neq
```

此时不能再取反。

------

## 九、问题八：PC 分支选择逻辑容易接错

### 现象

实现 `bner0` 时，需要让 PC 在 `PC+1` 和 `branch_addr` 之间选择，容易出现 MUX 接反或没有真正接入 PC 的问题。

------

### 正确结构

第一层选择：

```text
normal_pc_next = branch_taken ? branch_addr : PC + 1
```

第二层清零：

```text
PC_D = CLR ? 0000 : normal_pc_next
```

结构为：

```text
PC + 1 --------\
                MUX1 ---- normal_pc_next ----\
branch_addr ---/                              MUX2 ---- PC_reg.D
     ^                                           ^
     |                                           |
branch_taken                                    CLR
```

------

### 调试方法

执行到：

```text
PC = 6
instr = D1
```

时，接 Probe 观察：

```text
is_bne
neq
branch_taken
branch_addr[3:0]
PC_D[3:0]
```

正确结果应该是：

```text
is_bne = 1
neq = 1
branch_taken = 1
branch_addr = 0100
PC_D = 0100
```

如果：

```text
branch_taken = 1
PC_D = 0111
```

说明 PC 选择 MUX 接反或没有接对。

如果：

```text
PC_D = 0100
时钟后 PC 还是 7
```

说明 PC_reg.D 没有真正接到这个 MUX 输出，或者 PC_reg 的 EN/CLK 有问题。

------

## 十、问题九：branch_addr 字段容易接错

### 现象

`bner0` 的跳转地址容易误接成 `instr[3:0]`。

------

### 正确理解

`bner0` 格式为：

```text
11 addr rs2
```

其中：

```text
addr = instr[5:2]
rs2  = instr[1:0]
```

对于：

```text
D1 = 11010001
```

拆分为：

```text
11 0100 01
```

所以：

```text
branch_addr = 0100 = 4
```

如果误接为：

```text
instr[3:0]
```

得到的是：

```text
0001 = 1
```

就会跳转错误。

------

### 经验总结

每种指令的字段划分必须严格按照指令格式来，不能只凭低位字段猜。

------

## 十一、问题十：为什么最后 PC 停在 7

### 现象

完整程序运行结束后，最终状态为：

```text
PC = 7
r2 = 55
```

一开始疑惑：为什么 PC 不继续加到 8？

------

### 原因分析

地址 7 的指令是：

```text
7: bner0 r3, 7
```

机器码为：

```text
DF = 11011111
```

拆分为：

```text
opcode = 11
addr   = 0111
rs2    = 11
```

含义：

```text
if (r0 != r3)
    PC = 7;
else
    PC = PC + 1;
```

最终程序状态：

```text
r0 = 10
r3 = 1
```

所以：

```text
r0 != r3
```

条件一直成立。

因此每次执行 PC=7 的指令，都会让：

```text
PC_next = 7
```

而不是：

```text
PC_next = 8
```

------

### 经验总结

当前 sCPU 没有真正的 `halt` 停机指令。

所以程序用：

```text
7: bner0 r3, 7
```

制造一个自跳转死循环，模拟程序结束。

最终停在：

```text
PC = 7
```

是正确现象。

------

## 十二、整体经验总结

通过这些问题，可以总结出设计 sCPU 时需要重点注意：

1. 先保证基础模块依赖清晰，不要随便导入大型旧项目。
2. 寄存器输出红色 `E` 时，优先考虑是否还没有时钟写入。
3. 总线观察优先使用 Probe，不要每一位都拆开看。
4. 多位 Pin 可以设置 `Data Bits`，不需要多个单独 Pin。
5. 寄存器堆必须有写使能 `wen`。
6. `add` 需要两个读端口。
7. 分支指令最关键的是比较结果、跳转条件和 PC 选择。
8. 比较器输出含义必须先定义清楚。
9. 指令字段拆分必须严格按照指令格式。
10. 调试 CPU 时，要多观察中间信号，而不是只看最终结果。

调试 sCPU 时最有效的思路是：

```text
当前 PC 是多少？
当前 instr 是什么？
opcode 是什么？
各字段拆分是否正确？
控制信号是否正确？
数据通路结果是否正确？
PC_D 是否正确？
时钟后状态是否正确？
```

这和 F4 中的状态机思想一致：

```text
当前状态
↓
根据指令语义计算下一状态
↓
时钟上升沿更新状态
```
